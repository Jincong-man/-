# comments-check

> 锦聪的小店 — 代码注释检查技能

## 功能概述

这个技能对项目中的代码文件进行三项注释检查：

| 检查项 | 说明 | 触发词 |
|--------|------|--------|
| **注释覆盖率** | 检查注释是否够多：每个函数、每条核心逻辑都应有注释，目标 **2行注释 : 8行代码** | "检查注释"、"注释够不够"、"注释覆盖" |
| **注释匹配度** | 检查注释内容是否与代码实际逻辑一致，有没有"写A做B"的情况 | "注释对不对"、"注释匹配"、"注释是否准确" |
| **新手可读性** | 检查注释是否通俗易懂，没有跳步骤、没有未解释的术语 | "注释好不好懂"、"小白能看懂吗" |

---

## 检查工作的流程

对每个目标文件，按以下顺序检查：

### 第一步：注释覆盖率检查（数量）

1. 统计文件总行数（不含空行）
2. 统计注释行数（`//` 单行注释、`/* */` 多行注释、HTML `<!-- -->` 注释、CSS `/* */` 注释）
3. 计算注释比例 = 注释行数 ÷ 有效代码行数
4. **目标比例：约 20%**（每 10 行中约 2 行注释）

重点检查以下位置是否缺少注释：

| 必须加注释的位置 | 说明 | 示例 |
|---|---|---|
| 每个函数/方法 | 函数的作用、参数含义、返回值 | `/** 根据关键词过滤商品列表 */` |
| 复杂条件判断 | 这个 if 在判断什么、为什么这样写 | `// 购物车为空时不提交订单` |
| 数据转换逻辑 | 为什么做这个转换、转换前后格式 | `// 将价格从字符串转为数字，便于后续计算` |
| 魔法数字 | 解释这个数字的含义 | `// 最小密码长度为 6 位` |
| 正则表达式 | 解释正则在匹配什么 | `// 匹配中国大陆手机号格式` |
| CSS 关键布局 | 为什么用这个布局方式 | `/* Grid 布局实现响应式商品卡片 */` |
| HTML 重要区块 | 区块的用途 | `<!-- 商品分类区域，由 Vue 循环渲染 -->` |

---

### 第二步：注释匹配度检查（质量）

对每条注释逐一核对：

1. **读注释 → 看代码**：注释说这个函数做X，代码是否真的在做X？
2. **读代码 → 看注释**：这段代码的逻辑，注释有没有描述准确？
3. **检查过期注释**：代码改过了但注释没更新？

常见问题举例：

```
❌ 注释说"返回布尔值"，但代码返回的是字符串        → 注释与代码不符
❌ 注释说"校验邮箱格式"，但代码只检查了是否为空     → 注释夸大功能
❌ 函数逻辑已改为支持多条件，注释仍写"单一条件"     → 注释过期
✅ 注释准确描述函数做了什么，包括边界情况           → 匹配
```

---

### 第三步：新手可读性检查（视角）

以"计算机新人学生"的视角阅读每条注释，检查：

1. **术语是否解释**：出现"闭包"、"柯里化"、"防抖"等术语时，有没有用通俗语言解释？
2. **是否跳步骤**：注释假设读者已经知道某个前提，但这个前提并没有在代码或注释中出现？
3. **是否给出"为什么"**：注释不只是说"做了什么"，还要说"为什么这么做"？
4. **缩写是否展开**：`res`、`btn`、`cfg` 等缩写是否在首次出现时写全称？

新手友好 vs 不友好的对比：

```
❌ "使用 throttle 优化高频 DOM 操作"              → 术语未解释
✅ "使用节流（throttle：限制函数在短时间内只执行一次），避免频繁操作 DOM 导致页面卡顿"

❌ "reduce 计算 cartTotal"                       → 函数作用不明确
✅ "用数组的 reduce 方法（累加器：遍历数组，将每个元素的值累加到一个总数上），计算购物车中所有商品的总金额"

❌ "// 处理回调"                                  → 太笼统，没说处理什么
✅ "// 用户登录成功后，关闭登录弹窗并刷新商品列表"
```

## 输出格式

检查完成后，按文件分组输出报告：

```markdown
## 📊 注释检查报告 — Jincong.js

### 注释覆盖率：X%
- 总有效行：XXX
- 注释行：XX
- ⚠️ 未达到 20% 目标

### 🔴 缺少注释的位置
| 行号 | 问题 | 建议 |
|------|------|------|
| 68  | `filteredProducts()` 没有注释 | 添加：`/** 根据搜索关键词过滤商品，匹配名称、描述和分类 */` |

### 🟡 注释不匹配
| 行号 | 问题 | 现有注释 | 建议修改为 |
|------|------|----------|------------|

### 🟢 新手不易读懂的注释
| 行号 | 问题 | 现有注释 | 改进建议 |
|------|------|----------|----------|
```

## 工作约定

1. **先统计后细查**：先算出整体覆盖率，再逐条检查质量和可读性
2. **不修改代码**：只输出报告和建议，让用户自己决定是否修改
3. **优先级排序**：先报缺少注释的（功能缺失），再报注释写错的（会误导），最后报写得不够好的（改进空间）
4. **给出具体建议**：每条问题不只是说"不好"，还要给出改进后的注释版本
5. **按文件分别报告**：每个文件一份报告，方便用户逐个处理
